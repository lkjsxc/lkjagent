# YOLO Mode

## Purpose

Define the owner-visible product posture: lkjagent acts decisively inside its
configured container sandbox until honest completion, a real external block, a
runtime budget, an unrecoverable environment failure, or daemon stop.

## Contract

The sandbox boundary is the safety model. Inside that boundary, admitted
filesystem, shell, graph, memory, document, artifact, verification, recovery,
maintenance, and compaction actions do not require permission prompts.

The main loop is not plan mode. A plan, scaffold, README tree, manifest, graph
note, weak artifact path, recoverable fault, compaction pressure, or no-op
maintenance result is not terminal when useful admitted work remains.

## Terminal States

An owner task stops only when the central completion gate proves completion, a
concrete missing external owner input blocks progress, the turn budget writes a
structured partial handoff, the endpoint or environment is unrecoverable with
native tools, or the daemon stops.

## Owner Questions

The product asks only concrete external questions. It does not ask which
internal tool to use, whether to retry parse or schema faults, whether to
inspect, audit, repair placeholders, continue from weak paths, compact runtime
state, preempt maintenance, or refuse missing-evidence completion.

## Completion

YOLO mode forbids premature completion. `agent.done` is admitted only by the
central completion gate. Plans, notes, memory rows, file existence, directories,
README files, manifests, links, scaffolds, placeholder prose, old readiness
evidence, and model claims never satisfy completion by themselves.
